# Pythonカンバン型タスクマネジメントツール 要求仕様書

## 1. 概要

JTC（Japanese Traditional Company）環境で使える、数名規模チーム向けの軽量カンバン型タスクマネジメントツール。  
Pythonで実装し、ソフトウェアのインストールやサーバー構築が困難な環境での運用を前提とする。

---

## 2. 環境・制約条件

- **動作OS**: Windows（顧客企業貸与PC）
- **配布形式**: PyInstallerによる単一exeが理想（ポータブル運用）
- **ネットワーク制約**: ローカルサーバーの起動がWindowsポリシーでブロックされる可能性があるため、Webベース（ローカルホスト）アーキテクチャは採用しない
- **共有インフラ**: Samba共有フォルダを利用。新たなサーバー構築は不要
- **想定ユーザー規模**: 数名（目安3〜10名程度）

---

## 3. アーキテクチャ方針

### 3.1 UIフレームワーク

**PyQt6 / PySide6** を採用。

- ドラッグ&ドロップによるカード移動（QDrag/QDrop）
- アニメーション（QPropertyAnimation）
- ネイティブデスクトップアプリとして動作
- WebベースUIへの将来移行は考慮しない（必要になった時点でフルリライトを検討）
- ビジネスロジックとUIレイヤーは明確に分離しておく（将来の移植コスト低減のため）

### 3.2 データストレージ

**SQLite**（単一ファイル）を採用。

- Samba共有フォルダ上にSQLiteファイルを配置し、クライアントアプリから参照・更新する
- ルールデータと実務データは**別ファイル**として管理する（ライフサイクルが異なるため）
- WALモード有効化による並行読み取り性能の向上
- Samba越しのネットワーク遅延によるロックタイムアウトに備えた書き込みリトライ機構を実装する

### 3.3 依存ライブラリ方針

最小限に抑える。現時点での想定依存:

| ライブラリ | 用途 |
|---|---|
| PyQt6 または PySide6 | UI |
| sqlite3 | DBアクセス（Python標準ライブラリ） |
| plotly | ガントチャート出力 |
| jinja2 | テキストエクスポートテンプレート（将来機能） |

## 4. データ設計

### 4.1 ファイル構成

```
[Samba共有フォルダ]/
├── rules.db        # ルールデータ（カンバン定義・担当者・タグ定義など）
├── work.db         # 実務データ（チケット・進捗情報など）
└── manager.lock    # managerロール排他制御用ロックファイル
```

### 4.2 データの種類

#### ルールデータ（rules.db）
カンバンのマネジメントルールを定義する。チームのセットアップ時に構成し、運用中の変更頻度は低い。

- **マストデータ**（固定テーブル）: 担当者、カンバン定義
- **タグ**（ユーザー定義テーブル）: ユーザーがカスタマイズ可能な汎用データ定義

#### 実務データ（work.db）
ルールデータを元に日常業務で作成・更新されるデータ。チケット本体、進捗、タグの実体値など。

### 4.3 マストデータ

#### 担当者テーブル
- 担当者名、メールアドレス（詳細はスキーマ設計フェーズで定義）

#### カンバン定義テーブル
- 番号、名前、担当者、ステータス、開始日、終了予定日、備考（詳細はスキーマ設計フェーズで定義）

#### ステータス定義テーブル
カンバンボードの列に対応する値をルールデータとして管理する。

```
id: 1, name: "未着手",    order: 1
id: 2, name: "仕掛り中",  order: 2
id: 3, name: "完了",      order: 3
```

- カンバンボードの列はこのテーブルを読んで動的に生成する
- チケットのステータスフィールドはこのテーブルのidを外部キーで参照する
- チームごとにステータスの種類・順序・名称をカスタマイズ可能
- バリデーションは「ステータス定義テーブルに存在するidか」を確認するだけでよい
- プロンプト自動生成機能でも動的に利用可能なステータス一覧を埋め込める

**汎用Enum型への拡張について:**  
「優先度（高/中/低）」のような同様の仕組みが必要な項目が将来生じた場合に備え、ステータス定義テーブルは汎用化しやすい構造にしておく。ただし初期バージョンではステータス専用とし、汎用化は将来の拡張課題とする。

### 4.4 タグ設計

チケットに任意のkey-valueフィールドを追加できる仕組み。タグ定義はフラットなリストとして管理し、グループ構造や階層関係は持たない。

**タグ定義テーブル（概念）:**

| カラム | 内容 |
|---|---|
| id | タグ定義ID |
| name | タグ名（例: 機種名、会社名） |
| field_type | フィールド型（text / date 等） |

**設計方針:**
- タグはフラットなkey-valueとして定義する。グループ構造・階層構造は持たない
- 「機種名」「バージョン」「リリースマイルストーン」はそれぞれ独立したタグ定義として登録する
- 複数タグの組み合わせ検索はフィルター機能のAND指定で実現する

**タグ使用例①（製品開発チーム）:**
```
タグ定義: 機種名(text), バージョン(text), リリースマイルストーン(date), 機能名(text)
チケット例: #12 ビーム発射機能の設計 / 田中 / 仕掛り中 / 機種名:アルファ / バージョン:1.0 / 機能名:ビーム発射
```

**タグ使用例②（システム導入管理チーム）:**
```
タグ定義: 会社名(text), 顧客氏名(text)
チケット例: #334 パンケーキ焼きマシン導入 / 高橋 / 未着手 / 会社名:ベータ株式会社 / 顧客氏名:佐藤さん
```

## 5. ロール・アクセス制御

SQLiteはユーザー概念を持たないため、アプリ層で実装する。

| ロール | 読み取り | 書き込み | 同時接続 |
|---|---|---|---|
| manager | ○ | ○ | 1名のみ（先着優先） |
| member | ○ | × | 制限なし |

### manager排他制御の仕組み

- Samba上に `manager.lock` ファイルを設置
- managerがアプリを開く際にロックファイルに自分の情報＋タイムスタンプを書き込む
- 別のmanagerが接続しようとした場合、ロックファイルの存在を確認してブロック
- アプリ終了時にロックファイルを削除
- **クラッシュ対策**: ロックファイルにハートビート（定期タイムスタンプ更新）を実装し、一定時間更新がなければ無効なロックとみなす

### memberの動作

- ロックファイルの有無に関わらずデータを読み取り可能
- manager編集中の場合は画面に警告を表示する
  - 例: 「現在マネージャーがタスクを編集中です。今見ている情報は古い可能性があります」
- 書き込みUIはアプリ層で無効化（ボタン・フォームをdisable）

## 6. 機能要件

### 6.1 カンバンボード（必須）

- 全チケットのバックログ表示
- 列間のドラッグ&ドロップによるステータス変更
- カード移動時のアニメーション

### 6.2 フィルター機能（必須）

以下の条件でのチケット絞り込みが可能:
- タグごとの検索（AND / OR 指定可能）
- 開始日・終了予定日による範囲指定
- ステータスによる絞り込み
- 担当者による絞り込み

### 6.3 ガントチャート出力（必須）

- チケット情報をガントチャートとして出力
- ライブラリ: **plotly**（`plotly.express.timeline`）
- 出力形式: HTMLファイル（ブラウザで表示）
- カンバンボードと同様のフィルター機能を提供
- チャート上での直接編集は不要（表示専用）

### 6.4 テキストエクスポート（条件付き実装）

- カンバンボードの内容をテキストデータとして出力する機能
- **実装方針**:
  - 設計段階ではJinja2テンプレートによる汎用エクスポートの余地を残す（rules.dbにテンプレート保存カラムを用意）
  - 初期実装は開発者の現場フォーマットに特化したカスタム実装（トピックブランチで管理）
  - 汎用テンプレート機能は実装してみてUXが許容できる複雑さなら公開リリース、そうでなければトピックブランチに留める

### 6.5 外部ファイルからの進捗更新取り込み機能（manager限定）

進捗会議の文字起こしをAIに渡して進捗データを生成し、それをツールに取り込むことで手入力の手間を省く機能。

#### 想定ワークフロー

```
進捗会議 → 文字起こし → AIに進捗データを生成させる
→ 進捗データファイル（JSON）を保存 → 本ツールで取り込み
→ 差分プレビュー確認 → manager承認 → 実務データ更新
```

#### 進捗データのフォーマット

フォーマットは**JSON**を採用。スキーマバリデーションとの親和性が高く、AIによる安定した出力が期待できる。

```json
{
  "updates": [
    {
      "ticket_id": 12,
      "status": "仕掛り中",
      "assignee_id": 3,
      "end_date": "2026-02-28",
      "note": "設計レビュー完了"
    }
  ]
}
```

不明・言及なしのフィールドはAIに省略させる（ハルシネーション被害の局所化）。

#### フェイルセーフ設計（異常系制御）

**バリデーション層**（取り込み前にチェック・エラーは取り込み拒否）:

- 存在しないticket_idへの更新
- 定義外のステータス値
- 存在しない担当者ID
- 必須フィールドの欠損
- JSONスキーマ不正（パースエラー含む）

**プレビュー＆承認フロー**（最重要の安全弁）:

- 取り込み実行前に「何がどう変わるか」の差分をmanagerに提示
- manager が明示的にOKを押した場合のみDBに書き込む
- 変なデータが混入していても人間が目視確認できる

**トランザクション制御**:

- 複数チケットの一括更新は全件成功 or 全件ロールバックのどちらか
- 一部だけ書き込まれた中途半端な状態を作らない

#### フォーマット定義・プロンプト自動生成機能

ルールデータ（担当者・ステータス定義・タグ等）を元に以下の2つをアプリ上で表示・コピーできる:

**① 進捗データのフォーマット定義**
- AIやユーザーが参照できるJSONスキーマ
- ルールデータが変わると自動的に更新される

**② AIへのプロンプトテンプレート**
- 文字起こしを貼り付けて使えるプロンプトをルールデータから動的生成
- 利用可能なステータス・担当者ID・チケット一覧などを自動で埋め込む
- 「不明な場合はフィールドを省略」「存在しないticket_idは使用しない」などの安全指示を含む

生成プロンプトのイメージ:

```
以下の進捗会議の文字起こしから、各タスクの進捗情報を
指定のJSONフォーマットで出力してください。

【利用可能なステータス】
未着手 / 仕掛り中 / 完了 / 保留

【利用可能な担当者】
田中(id:1) / 高橋(id:2) / 鈴木(id:3)

【出力フォーマット】
（フォーマット定義を自動埋め込み）

【注意事項】
- 不明・言及なしの項目はフィールドを省略してください
- 存在しないticket_idは絶対に使用しないでください
- ステータスは上記リストの値のみ使用してください
```

## 7. 非機能要件・将来対応

### 7.1 変更ログ・履歴（将来機能）

- 初期バージョンでは実装しない
- ただし全テーブルに `created_at`、`updated_at` カラムを最初から設けておき、後からトリガーベースの変更ログを追加しやすくする

### 7.2 公開・配布

- OSSとしての公開を検討中
- テキストエクスポートの汎用テンプレート機能は実装品質次第でリリース判断
